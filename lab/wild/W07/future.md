# W-7 シリーズ — 将来展望・応用可能性の議論

本ドキュメントは W-7 / W-7-2 実験シリーズの結果を踏まえた応用可能性・将来方向性についての議論を記録する。

---

## 1. W-7-2 の達成とその意味

W-7-2 Phase 3（LKB + SRM）で CV R²=0.934，P@20=0.900 を達成し，
従来の Burau/3-Stage 天井（R²≈0.27，P@50=0.868）を大きく突破した。

**重要な知見**：R²≈0.27 の天井は表現の非忠実性（Burau が K≥5 で非忠実）が原因ではなく，
忠実な LKB 表現もスカラー圧縮すると R²=0.206 とむしろ低下した。
真の原因は「行列表現をスカラー 1 個に圧縮する」情報損失であり，
SRM（標準表現行列）の 25 次元ベクトルを保持することで突破できた。

この結果は「**どの順序が良いかを，実際に試さずに事前予測できる**」という実用的スクリーニングツールの可能性を示す。

## 2. トポロジー解析における多変量解析の位置づけ

### 古典的代数トポロジー
純粋数学の文脈では多変量統計解析はほぼ使われない。ホモロジー群・表現論の証明が主ツール。

### TDA（Topological Data Analysis）
「トポロジーが特徴量を供給し，統計・ML が予測する」という分業が定着。
本シリーズの Burau trace や SRM もこの観点では「位相的特徴量の供給源」のひとつ。

### 本シリーズの位置づけ
```
古典トポロジー  → 証明・分類（目的が異なる）
TDA            → データ → 位相特徴量 → ML（供給方向が逆）
W-7 シリーズ   → 位相不変量（表現行列）→ 多変量回帰で有効性診断
```

「どの表現がダイナミクスの dist を線形近似できるか」という表現論的選択問題であり，
純トポロジーより実験計画・最適化の方法論に近い貢献。

## 3. 医学への転用の限界

### SRM スクリーニングが医学に向かない本質的理由

SRM+Ridge が「90% の順序を除外できる」という性質は，臨床文脈では問題になる：

- **Precision Medicine との矛盾**：「平均で悪い順序」にこそ特定サブグループ（遺伝子型・病態）で奏効率が高い組み合わせが潜む可能性がある
- **倫理的・規制的問題**：事前除外には患者レベルでの正当化根拠が必要
- **個体差**：医学では患者ごとの異質性が支配的で，群平均の dist が個々の結果を代表できない

**Cox 回帰との関係**：患者レベルのアウトカム（生存・再発）には Cox 回帰が標準。
遺伝子発現・血液マーカー等の biomarker と SRM を組み合わせることは可能だが，
その場合トポロジカル変量は「特徴量の供給源のひとつ」に格下げされる。

### フレームワークが医学で使えるとすれば

```
Phase 0/I : どの順序を試す価値があるか（候補生成）← トポロジーが貢献
Phase II/III: その順序で誰が効くか（患者選択）  ← Cox回帰 + biomarker が主役
```

前段階のスクリーニングとして位置づけるなら意味があるが，
汎化検証と Cox との比較が査読では求められる可能性が高い。

## 4. ソフトウェア工学への応用：CPM との相補性

### Critical Path Method（CPM）との分業

| 手法 | 答える問い |
|---|---|
| CPM | 依存関係を守ると最短で何日かかるか |
| 本FW | **依存がない（自由な）タスク群の中でどの順序が最終状態を安定させるか** |

CPM は依存グラフ（DAG）の構造を扱うが，**依存のない辺の間の順序効果は考慮しない**。
本フレームワークはその空白を埋める。

### 対応表（ソフトウェア工学文脈）

| 数学的概念 | ソフトウェア工学への読み替え |
|---|---|
| 状態空間 P | コードベースの状態（技術的負債，API 契約，結合度） |
| 介入 E_i | 機能 i の実装 |
| 時間発展 U_t | CI/CD・統合テスト・エントロピー蓄積 |
| dist | 統合後バグ密度，リファクタリングコスト，テスト失敗累積数 |
| 参照順序 | ベテランが経験的に選ぶ順序 |

非可換性の実例：
```
E_auth ∘ E_user ≠ E_user ∘ E_auth
```
認証サービスを先に実装すると，ユーザーサービスのモックインターフェースが認証前提に引きずられる。

## 5. AI 自動プログラミング × マイクロサービスへの応用

### なぜ最も適合するドメインか

| フレームワークの要求 | マイクロサービス + CI/CD での充足状況 |
|---|---|
| 演算子 E_i が独立 | インターフェース契約で分離されている |
| 状態空間 P が定義可能 | テスト pass/fail が定量的状態 |
| dist が測定可能 | **CI パイプラインが自動計測** |
| 個体差がない | AI エージェントは再現性が高い |
| 試行錯誤できる | git + revert で可逆，フィードバックが高速 |

### AI が「順序を誤る」問題との対応

AI コーディングエージェントが「そこを外したら動かない」実装をしてしまう現象は，
非可換性の見落としそのものである。MCP はコードベースの現在状態（P の読み取り）を改善するが，
「次にどのモジュールを実装すべきか」（S_K 上の最適化）には答えない：

```
MCP  → 現在の P を正確に把握する（状態の認識）
本FW → 次にどの E_i を適用すべきか決める（順序の最適化）
```

両者は相補的なレイヤーを形成する。

### 実用パイプライン

```
① 要件定義 → K 個のマイクロモジュールに分解
② 依存グラフ解析（MCP/静的解析）→ ハード依存を除去
③ 残った自由順序に SRM スコアリング → 推奨順序を生成
④ AI エージェントが推奨順序で実装
⑤ 各モジュール後に CI → dist をリアルタイム計測
⑥ ⑤のデータが次プロジェクトのモデル更新に使われる
```

プロジェクトを重ねるごとにモデルが精度を上げる学習サイクルが成立する。

### マイクロサービス設計原則との偶然の一致

| 設計原則 | フレームワークへの効果 |
|---|---|
| Single Responsibility | E_i が純粋な演算子に近づく |
| Interface Segregation | 演算子間の結合が最小化 |
| テスト駆動開発 | 各 E_i 適用後の P の変化が定量測定可能 |

設計思想レベルでフレームワークが動く条件を自然に満たしている。

### 医学との比較

| 問題点 | 医学 | AI 自動プログラミング |
|---|---|---|
| 個体差 | 患者ごとに大きく異なる | **同じコードベース → 高再現性** |
| 除外のリスク | 倫理的・規制的に問題 | **スプリント計画の変更だけ** |
| フィードバック速度 | 数年単位 | **CI/CD で数時間〜数日** |
| 可逆性 | 低い | **git で完全可逆** |

## 6. 残る開問・将来実験候補

### 理論的課題

1. **LKB 行列要素全展開**：225 次元をすべて特徴量にすると R² はどこまで上がるか
2. **SRM R²=0.93 の残余 7%**：どのような置換・演算子特性で説明されるか
3. **確率的演算子への拡張**：AI エージェントの LLM サンプリングは E_i に確率的ゆらぎを与える；決定論的仮定の緩和が必要
4. **K→∞ スケーリング**：SRM 次元 (K-1)² ∝ K² 増大時の R² 挙動

### 応用検証候補

1. **ソフトウェアプロジェクトへの実測**：実際のマイクロサービス開発で dist（CI 失敗累積）を計測し，SRM スクリーニングの有効性を検証
2. **AI エージェントの実装順序比較**：同一タスクを異なる順序で AI に実装させ，dist の分布を測定
3. **CPM との統合**：DAG の依存グラフから「自由な辺」を自動抽出し SRM スクリーニングを適用するツール化

## 7. 論文貢献の定位

本フレームワークの最も誠実な貢献の主張：

> 「介入順序空間（S_K）の数学的構造を群論・表現論で特徴づけ，
> **個体差が小さく dist が測定可能なドメイン**（工学的プロセス最適化，AI 自動開発）において
> 全 K! 通りの試行なしに有望な順序を事前スクリーニングできることを実証した。」

臨床応用については「患者レベルの異質性を含む場合は追加検証が必要」と明示し，
スコープを工学的最適化問題に絞った方が論文として強くなる可能性がある。

査読対応として想定される指摘：
- 「医学への適用可能性は？」→ 上記の限界を先回りして記載
- 「汎化は？」→ Exp2（ジェネレータ堅牢性）・Exp3（次元堅牢性）が根拠
- 「omics アプローチと何が違うのか？」→ 患者データ不要・数学的根拠・設計段階での使用可能性
